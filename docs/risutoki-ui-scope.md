# RisuToki UI 기준 봇 편집 범위

## 1. 이 문서의 목적

이 문서는 1차 봇 editable 범위를 정할 때,
`RisuToki`가 실제로 보여 주는 편집 단위를 기준으로 무엇을 가져오고 무엇을 미루는지 정리한 문서입니다.

핵심 원칙은 단순합니다.

> `RisuToki`가 실제로 만지는 범위 중, 모듈 툴과 겹치지 않는 카드 본체 영역만 1차에 넣는다.

---

## 2. `RisuToki`에서 보이는 큰 편집 단위

`RisuToki` UI에서 확인되는 큰 단위는 아래와 같습니다.

- 이름
- 설명
- 첫 메시지
- 글로벌노트
- 기본변수
- CSS
- Lua
- 로어북
- 정규식
- 에셋

이 중 1차 범위는 카드 본체와 에셋 보존입니다.
현재 구현에서는 embedded `module.risum`도 같이 분해합니다.

---

## 3. 1차 editable에 넣는 항목

현재 직접 편집 파일로 뽑는 항목은 아래와 같습니다.

- `name`
- `description`
- `firstMessage`
- `additionalFirstMessages`
- `globalNote`
- `css`
- `defaultVariables`

이유는 아래와 같습니다.

- `RisuToki`에서 실제로 자주 만지는 카드 본체 필드에 가깝다
- 모듈 툴과 직접 겹치지 않는다
- roundtrip 안정성을 해치지 않고 좁게 유지할 수 있다

---

## 4. 보존은 하지만 1차 editable에서는 제외하는 항목

아래 항목은 원본 카드 JSON에는 남겨 두지만,
1차 editable 범위에는 넣지 않습니다.

- `personality`
- `scenario`
- `creatorcomment`
- `tags`
- `lowLevelAccess`

이유는 아래와 같습니다.

- 존재는 하지만 1차 핵심 편집면으로 보기에는 범위가 넓다
- 일부는 `RisuAI` 본체에는 있어도 `RisuToki`의 좁은 편집면 기준에서는 우선순위가 낮다
- 1차 목표가 "필드 확장"이 아니라 "지원 포맷 확장"이기 때문이다

---

## 5. 모듈 툴과 겹쳐서 제외하는 항목

아래 항목은 `RisuToki`에 보이더라도 1차 봇 전용 범위에서는 제외합니다.

- Lua
- 로어북
- 정규식

이유:

- 실질적으로 embedded `module.risum` 쪽 데이터와 연결된다
- 기존 `risuai-module-tools`가 맡는 영역과 겹친다

즉 1차에서는:

> 보이더라도 새 봇 편집 기능으로 다시 만들지 않는다

를 원칙으로 둡니다.

---

## 6. 에셋 처리 위치

에셋은 1차에서도 다룹니다.

다만 성격은 텍스트 editable과 다릅니다.

- 추출한다
- 사람이 보기 좋은 파일명으로 저장할 수 있다
- build 때 다시 넣는다
- 기본은 Git 비추적 대상으로 둔다

즉 에셋은 **편집 대상이라기보다 보존 및 재조합 대상**에 가깝습니다.

---

## 7. `lowLevelAccess` 정리

`lowLevelAccess`는 RisuAI 본체에서는 고급 설정입니다.

하지만 현재 1차 기준에서는 editable 대상에 넣지 않습니다.

이유:

- `RisuToki`의 좁은 편집면 기준에서 핵심 범위가 아님
- 트리거/Lua/CBS 같은 더 넓은 영역과 연결된다
- 기존 모듈 툴 수준을 넘는 기능 확장으로 번질 수 있다

따라서 현재는 원본 보존만 합니다.

---

## 8. 한 줄 요약

현재 1차 봇 범위는 `RisuToki` 기준에서 **이름, 설명, 첫 메시지, 추가 첫 메시지, 글로벌노트, 기본변수, CSS**를 직접 편집 대상으로 두고,
Lua / 로어북 / 정규식은 embedded `module.risum`을 풀어서 기존 모듈 툴 영역으로 넘기는 방식입니다.
