# 설계 방향 문서

## 1. 이 프로젝트의 1차 목표

이 프로젝트의 1차 목표는 **새 기능을 많이 만드는 것**이 아닙니다.

핵심 목표는 아래 한 줄로 정리할 수 있습니다.

> `risuai-module-tools`가 하던 수준의 `extract / build` 흐름을, 사용 가능한 다른 RisuAI 포맷에도 확장한다.

즉 기준은 항상 기존 도구입니다.

- 모듈은 `risuai-module-tools`
- 봇은 `RisuToki`가 실제로 만지는 범위

를 참고합니다.

---

## 2. 왜 이렇게 범위를 좁히는가

RisuAI 포맷은 구조가 넓고 복잡합니다.

- 카드 본문 필드가 많음
- 로어북, 정규식, Lua, CSS가 섞일 수 있음
- 에셋도 이미지뿐 아니라 오디오, 비디오가 들어갈 수 있음
- 봇 포맷도 `charx`, `png`, `jpg`, `jpeg`처럼 여러 컨테이너가 있음

처음부터 모든 필드를 다 편집 대상으로 잡으면:

- 구현이 커지고
- roundtrip 안정성이 떨어지고
- 기존 도구와 책임이 겹치고
- 실제로 자주 안 쓰는 필드까지 관리해야 해서 불편해집니다.

그래서 1차에서는 **기존 도구가 이미 하던 수준을 기준으로**, 자주 쓰는 범위만 먼저 다룹니다.

---

## 3. 현재 범위

### 3.1 모듈

모듈은 새로 설계하지 않습니다.

- `.risum` unpack / pack
- `module-vcs` 스타일 소스 분해 / 조립

를 그대로 기준으로 둡니다.

즉, 1차에서 모듈은 **새 개념을 발명하는 대상이 아니라 기존 워크플로를 유지하는 대상**입니다.

다만 로어북 분해는 안정성을 위해 한 가지 원칙을 둡니다.

- `triggerlua`, `backgroundEmbedding`처럼 원래 구조에서 역할이 분명한 항목만 별도 source 파일로 분해
- 로어북 엔트리 `content`는 내용만 보고 Lua/CSS로 추측하지 않고 기본적으로 텍스트로 유지

### 3.2 봇

봇은 `RisuToki` 기준에서 카드 본체와 embedded `module.risum`을 함께 다룹니다.

- 카드 본체는 `src/card/` 아래의 텍스트 파일로 편집
- `module.risum`은 `src/module/` 하위 모듈 프로젝트로 분해

현재 1차 editable 범위:

- 이름
- 설명
- 첫 메시지
- 추가 첫 메시지들
- 글로벌노트
- 기본변수
- CSS

로어북, 정규식, Lua는 봇 전용 편집 파일로 따로 만들지 않고,
embedded `module.risum`을 풀어서 모듈 쪽 편집 구조로 넘깁니다.

현재 1차에서 직접 카드 editable 대상으로 삼지 않는 것:

- `lowLevelAccess`
- 기타 드물게 쓰는 고급 카드 필드

### 3.3 에셋

에셋은 1차에서도 중요합니다.

그래서:

- 추출한다
- build 때 다시 넣는다
- 사람 보기 좋은 파일명으로 저장할 수 있다
- 하지만 기본은 Git 추적 대상에서 제외한다

를 원칙으로 둡니다.

---

## 4. 왜 Git 친화적이어야 하나

RisuAI 파일은 원본 그대로 두면 한 파일 안에 정보가 너무 많이 들어 있습니다.

그 상태로는:

- 무엇이 바뀌었는지 보기 어렵고
- IDE에서 부분 수정하기 어렵고
- AI 에이전트가 안전하게 만지기 어렵습니다.

그래서 목표는 단순합니다.

- 사람이 읽기 쉬운 파일 구조로 나눈다
- Git diff가 읽히게 만든다
- 다시 원본 포맷으로 조립 가능하게 만든다

즉, RisuAI 파일을 **개발 가능한 소스 프로젝트처럼 다루는 것**이 목표입니다.

---

## 5. 에셋 처리 원칙

에셋은 봇과 모듈에서 모두 다루지만, 재조합 기준은 서로 다릅니다.

- 모듈: 순서 기반
- ZIP형 봇: 원래 경로 기반
- PNG형 봇: 청크 키 / 에셋 인덱스 기반

그래서 포맷별 재조합 규칙은 따로 유지해야 합니다.

대신 아래 저수준 로직은 공용화할 수 있습니다.

- 시그니처 기반 확장자 추정
- 미디어 종류 추정
- 표시용 파일명 정리
- 파일명 충돌 회피
- 원래 식별자와 표시용 경로 분리

즉 구조는:

> 공용 에셋 로직 + 포맷별 어댑터

입니다.

이 방식이면 기존 도구 수준을 넘는 새로운 사용자 기능을 크게 만들지 않으면서도,
확장자 불일치 같은 실제 문제는 더 안전하게 막을 수 있습니다.

---

## 6. 이번 단계에서 하지 않는 것

아래 항목은 1차 목표 바깥입니다.

- `risup`
- 로어북 전용 검색/트리/폴더 관리 기능
- 기존 도구에 없는 새로운 편집 단위 추가
- 카드/모듈의 모든 필드 지원

즉, 1차는 **지원 포맷 확장**이 중심이지, **기능 확장**이 중심이 아닙니다.

---

## 7. 구현 원칙

이번 단계에서는 아래 원칙을 지킵니다.

- 기존 도구의 책임 범위를 함부로 넘지 않는다
- roundtrip 안정성을 우선한다
- 편집 파일(`src/`)과 패킹 메타(`pack/`)를 분리한다
- 편집 가능한 범위는 좁고 명확하게 둔다
- 에셋은 추출하되 기본 Git 추적 대상에서는 뺀다
- standalone `.risum`과 embedded `module.risum`을 같은 파일이라고 가정하지 않는다
- 사용자 기능보다 내부 검증과 구조 정리를 우선한다

---

## 8. 한 줄 요약

이 프로젝트의 1차 목표는 `risuai-module-tools`와 `RisuToki`를 기준으로,
기존 도구가 하던 수준의 `extract / build` 워크플로를 다른 RisuAI 포맷에도 안전하게 확장하는 것입니다.
